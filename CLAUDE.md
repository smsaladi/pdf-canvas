# PDF Canvas — Project Instructions

## What This Is

PDF Canvas is a fully client-side, browser-based PDF editor built on MuPDF WASM. Users can open a PDF, visually select/move/resize annotations and form widgets, edit properties, create new annotations, and save — all with zero server round-trips.

## Key Documents

- `ARCHITECTURE.md` — Technical specification, architecture, detailed design, phased roadmap, success criteria
- `RESEARCH.md` — Landscape analysis of existing open-source PDF editor projects, what they do well, what they lack, and lessons learned
- `MUPDF_API_REFERENCE.md` — Comprehensive MuPDF JavaScript/WASM API reference for annotations, widgets, rendering, and document operations

Read all three before starting implementation.

## Technology Stack

- **PDF Engine**: MuPDF WASM (`npm i mupdf`) — AGPL licensed
- **Bundler**: Vite
- **Language**: TypeScript
- **UI**: Vanilla DOM / minimal framework (Preact acceptable if needed)
- **Threading**: Web Worker for all MuPDF operations
- **No other production dependencies** — no PDF.js, no pdf-lib, no Fabric.js

## Architecture Summary

```
Main Thread                          Web Worker
┌─────────────────────┐              ┌──────────────────┐
│ UI Shell             │   postMsg    │ MuPDF WASM       │
│  ├─ Toolbar          │◄───────────►│  ├─ Parse PDF    │
│  ├─ Viewport/Canvas  │             │  ├─ Render pages │
│  ├─ Overlay Layer    │             │  ├─ Annot CRUD   │
│  ├─ Properties Panel │             │  ├─ Widget CRUD  │
│  └─ Undo/Redo       │             │  └─ Save PDF     │
└─────────────────────┘              └──────────────────┘
```

All MuPDF calls happen in the worker. The main thread only handles DOM, user interaction, and coordinate conversion. Communication is message-based RPC with Transferable ArrayBuffers for performance.

## Critical Implementation Notes

1. **MuPDF is ESM-only**: `import mupdf from "mupdf"` — no CommonJS require.

2. **WASM in Web Worker**: MuPDF WASM must load inside a Web Worker. Vite needs configuration for WASM + Worker support. The `mupdf` npm package includes the WASM binary — ensure Vite copies it to the output directory.

3. **Coordinate system**: MuPDF uses top-left origin (same as screen). No Y-axis flip needed. But you must track `scale` and page offsets to convert between screen pixels and PDF points.

4. **Annotation appearance auto-regeneration**: After calling `annot.setRect()`, `annot.setColor()`, etc., MuPDF automatically flags the annotation for appearance stream regeneration. Call `annot.update()` or just re-render the page — you do NOT need to manually construct appearance streams.

5. **Incremental save**: `doc.saveToBuffer("incremental")` produces a minimal diff appended to the original PDF. This is fast and preserves digital signatures on unmodified content. Use this by default.

6. **Comment annotations**: The popup text for ALL annotation types is stored on the parent annotation via `getContents()` / `setContents()`. Popup annotations are display-only containers. Reply chains use IRT (In Reply To) references between annotation objects.

7. **Text markup annotations (Highlight, Underline, StrikeOut, Squiggly)**: These use QuadPoints, not Rect, for positioning. To move them, shift all QuadPoint coordinates by the drag delta. MuPDF auto-updates the parent Rect.

8. **Form widgets are separate from annotations**: Use `page.getWidgets()` not `page.getAnnotations()` for form fields. They share a similar API but are enumerated separately.

9. **Re-render strategy**: After any annotation mutation, re-render the entire affected page. MuPDF renders page + annotations as a single bitmap. Don't try to composite them separately. At 150 DPI for letter-size, that's ~1275×1650 pixels — renders in <100ms in WASM.

10. **Transferable objects**: When sending ArrayBuffers between main thread and worker, use the transfer list in `postMessage` to avoid copying: `worker.postMessage({ type: 'open', data: buffer }, [buffer])`.

### What To Test At Each Phase

**Phase 1 (Foundation):**
- Unit: `coords.ts` — PDF↔screen coordinate conversion with various scales and offsets
- Unit: `worker-rpc.ts` — message serialization/deserialization
- Integration: Worker can load MuPDF WASM, open a test PDF, return page count
- Integration: Worker renders a page and returns an ImageBitmap with correct dimensions
- E2E: Drop a PDF on the page → all pages render → scroll works → zoom changes re-render at correct size

**Phase 2 (Annotation Display & Selection):**
- Integration: Worker returns correct annotation DTOs for a PDF with known annotations (test fixture)
- Integration: Annotation types, rects, colors, contents all match expected values from fixture
- Unit: Overlay positioning — given a known rect and scale, overlay div has correct CSS position/size
- E2E: Open fixture PDF → annotation overlays visible → click annotation → selection box appears → properties panel shows correct data (type, color, contents, author)

**Phase 3 (Manipulation):**
- Integration: `setAnnotRect()` round-trip — set rect via worker, re-read it, verify it changed
- Integration: `setAnnotColor()`, `setAnnotContents()`, `setAnnotOpacity()` — same round-trip pattern
- Integration: Move highlight by shifting QuadPoints — verify all points shifted by correct delta and Rect auto-updated
- Integration: Delete annotation — verify it's gone on re-enumeration
- Unit: Undo stack — push, undo, redo, verify state at each step
- Unit: Undo stack — new action after undo clears redo stack
- E2E: Drag annotation → release → save → reopen saved PDF → annotation is at new position
- E2E: Change color via properties panel → save → reopen → color persists
- E2E: Delete annotation → save → reopen → annotation gone
- E2E: Ctrl+Z undoes last move, Ctrl+Shift+Z redoes it

**Phase 4 (Creation):**
- Integration: `createAnnotation("FreeText")` → set properties → verify it exists in annotation list
- Integration: Create each supported type, verify getType() matches
- E2E: Select FreeText tool → click+drag on canvas → annotation appears → can type text → appears in properties panel
- E2E: Select Note tool → click → sticky note icon appears → properties panel opens for comment entry
- E2E: Select Highlight tool → click+drag → highlight region created
- E2E: Create annotation → save → reopen in external viewer (check fixture output)

**Phase 5 (Widgets & Save):**
- Integration: Worker returns correct widget DTOs for a PDF with known form fields
- Integration: Widget move via setRect — round-trip verification
- Integration: `saveToBuffer("incremental")` produces valid PDF (non-zero bytes, starts with %PDF)
- E2E: Open form PDF → widget overlays visible → drag widget → save → reopen → widget at new position
- E2E: Save button produces downloadable file → file opens in browser's built-in PDF viewer

**Phase 6 (Polish):**
- E2E: Keyboard shortcuts — Delete key removes selected annotation, Ctrl+Z undoes, Ctrl+S saves
- E2E: Arrow keys nudge selected annotation by 1pt
- E2E: Multi-select with Shift+click → drag moves all selected

### Test Fixtures

**Create test fixture PDFs programmatically** using MuPDF in a Node.js script (`scripts/create-fixtures.ts`). This ensures fixtures have known, deterministic content:

```typescript
// scripts/create-fixtures.ts
import mupdf from "mupdf";

// Create a PDF with known annotations for testing
const doc = new mupdf.PDFDocument();
doc.insertPage(0, doc.addPage([0, 0, 612, 792], 0, null, ""));

const page = doc.loadPage(0);

// Add a FreeText annotation at a known position
const freetext = page.createAnnotation("FreeText");
freetext.setRect([100, 100, 300, 150]);
freetext.setContents("Test FreeText");
freetext.setDefaultAppearance("Helv", 12, [0, 0, 0]);

// Add a sticky note
const note = page.createAnnotation("Text");
note.setRect([400, 200, 424, 224]);
note.setContents("Test comment");
note.setColor([1, 1, 0]);
note.setIcon("Note");

// Add a highlight
const highlight = page.createAnnotation("Highlight");
highlight.setColor([1, 1, 0]);
highlight.setOpacity(0.5);
highlight.addQuadPoint([100, 300, 400, 300, 100, 315, 400, 315]);
highlight.setContents("Highlighted text comment");

// Save
const buf = doc.saveToBuffer("compress");
fs.writeFileSync("tests/fixtures/with-annotations.pdf", buf.asUint8Array());
```

Run `npx tsx scripts/create-fixtures.ts` to regenerate fixtures. Commit the generated PDFs to the repo.

### Round-Trip Verification

The most important test pattern: **open → mutate → save → reopen → verify**.

```typescript
// tests/integration/round-trip.test.ts
import { test, expect } from "vitest";
import mupdf from "mupdf";
import fs from "fs";

test("moving an annotation persists through save/reload", () => {
  // Open
  const bytes = fs.readFileSync("tests/fixtures/with-annotations.pdf");
  const doc = mupdf.PDFDocument.openDocument(bytes.buffer, "application/pdf");
  const page = doc.loadPage(0);
  const annots = page.getAnnotations();

  // Find the FreeText annotation
  const freetext = annots.find(a => a.getType() === "FreeText");
  expect(freetext).toBeDefined();
  const originalRect = freetext.getRect();

  // Mutate — move it 50 points right and 30 points down
  const newRect = [
    originalRect[0] + 50,
    originalRect[1] + 30,
    originalRect[2] + 50,
    originalRect[3] + 30,
  ];
  freetext.setRect(newRect);

  // Save
  const savedBuf = doc.saveToBuffer("incremental");
  const savedBytes = savedBuf.asUint8Array();

  // Reopen
  const doc2 = mupdf.PDFDocument.openDocument(savedBytes.buffer, "application/pdf");
  const page2 = doc2.loadPage(0);
  const annots2 = page2.getAnnotations();
  const freetext2 = annots2.find(a => a.getType() === "FreeText");

  // Verify
  const reloadedRect = freetext2.getRect();
  expect(reloadedRect[0]).toBeCloseTo(newRect[0], 1);
  expect(reloadedRect[1]).toBeCloseTo(newRect[1], 1);
  expect(reloadedRect[2]).toBeCloseTo(newRect[2], 1);
  expect(reloadedRect[3]).toBeCloseTo(newRect[3], 1);
});
```

### Rules

1. **Write tests BEFORE or ALONGSIDE implementation, not after.** If a function is added, its test is added in the same commit.
2. **Every worker RPC message type must have an integration test** that verifies the round-trip through the worker.
3. **Every user-facing interaction must have an E2E test** that simulates the actual click/drag/type sequence in a real browser.
4. **Fixture PDFs are deterministic.** Generate them with a script, commit them. Never manually create test fixtures.
5. **CI should run `npm test` and `npm run test:e2e`** — both must pass before any phase is considered complete.
6. **Coverage target**: 80%+ for `src/` files (excluding `worker.ts` which is hard to unit-test but covered by integration tests).
7. **Every bug fix MUST include a regression test.** When you encounter and fix a bug, write a test case (unit or integration) that would have caught the bug. This prevents the same bug from recurring. The test should fail without the fix and pass with it.
8. **User-reported bugs → fix + test in the same commit.** When the user reports a bug (including edge cases with specific PDFs), the workflow is: (a) diagnose the root cause, (b) create a minimal test fixture in `scripts/create-fixtures.ts` that reproduces the encoding/structure pattern (NOT the user's actual PDF), (c) write a test that fails without the fix, (d) implement the fix, (e) verify the test passes, (f) commit fix + test together. The test fixture should capture the *structural pattern* (e.g., Type0/Identity-H encoding, negative CTM values, multiple cm operators) rather than the specific document content.
9. **Test fixtures cover encoding diversity.** Maintain fixtures for each PDF encoding family encountered: WinAnsi/MacRoman (literal strings), Type0/Identity-H (hex glyph IDs), TJ arrays with kerning, multi-font content streams, and negative-CTM image placement. When a new encoding pattern causes a bug, add a fixture for that pattern.
